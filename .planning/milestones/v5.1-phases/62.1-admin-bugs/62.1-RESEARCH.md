## Summary

This phase is a pure UI/UX cleanup pass on the admin OverviewPage. No new features or backend data changes are needed — only renaming labels, removing redundant cards, and reordering the card layout. The codebase investigation reveals exactly which cards exist, how they map to backend data sources, and what is duplicated.

The key finding is that `UnmetDemandCard` and `TopZeroResultsCard` are **two separate cards pulling from the same backend endpoint** (`/events/demand`). They are genuine duplicates with only different titles and period awareness. The solution is to merge them into one `ZeroResultQueriesCard` that respects the period toggle. The "Total Searches" / "Gaps" / "New Leads" stat row (top stat mini-cards) also uses the term "Gaps" in the label — this must be renamed to "Zero-Result Queries".

The "Recent Explore Searches" card is the sole search card — no separate tag/filter/bar search cards exist. Search source separation is already only stored as metadata (`active_tags` field in payload), not surfaced in separate UI cards. So no merge is needed for searches, just a rename from "Recent Explore Searches" to "Recent Searches".

**Primary recommendation:** All changes are confined to `OverviewPage.tsx` (component renames, card titles, grid layout reorder) with a minor stat label rename. Backend endpoints, data shapes, and types.ts do not need changes for this phase.

### Key Data Source Investigation Findings

**Zero-Result Queries vs Gaps — are they the same source?**

- `TopZeroResultsCard`: calls `/events/demand` (user_events table, result_count=0 filter)
- `UnmetDemandCard`: calls `/events/demand` (same endpoint, same table, same filter)
- `StatCard "Gaps"`: uses `stats.gap_count` from `/stats` endpoint (Conversation table, top_match_score < threshold)

These are **two different data sources**:
- `/events/demand` = zero-result searches logged via the explore frontend (user_events table)
- `gap_count` from `/stats` = chat conversations where AI couldn't match (Conversation table)

The stat card "Gaps" and the `/events/demand` cards measure different things conceptually but per user decision, both rename to "Zero-Result Queries". The stat label change is cosmetic — do not change the underlying data source.

**Leads vs Newsletter Leads — are they the same source?**

- `RecentLeadsCard`: calls `/leads` (Conversation table, grouped by email)
- There is NO newsletter leads card on OverviewPage currently
- `NewsletterSubscriber` model exists but is only surfaced on DataPage/LeadsPage, not OverviewPage

Conclusion: No duplicate "newsletter leads" card exists on OverviewPage. The "Leads" rename is simply a label clarity fix. No merge needed for this card.

**Search type separation — tag/bar/filter:**

- `RecentExploreSearchesCard` displays all `search_query` events from user_events table
- The card already shows all three types (tag, text, browse) from a single backend call
- Source distinction is displayed inline in the existing card UI (tag chips, italic "Browse", text)
- No separate cards for tag vs bar vs filter exist
- Per decision: keep source as metadata, don't surface as separate categories → just rename the card

**UnmetDemandCard vs TopZeroResultsCard — confirmed duplicate:**

Both call `adminFetch<DemandResponse>('/events/demand', { days, page: 0, page_size: 5 })`. The only differences:
- `TopZeroResultsCard`: hardcoded `days: 30`, no period toggle awareness
- `UnmetDemandCard`: uses the `days` prop (period toggle aware), has an icon, uses numbered ranking rows

Merge strategy: Keep `UnmetDemandCard`'s structure (period-aware, ranked list, icon) but:
- Rename to `ZeroResultQueriesCard`
- Change title to "Zero-Result Queries"
- Use `AlertCircle` icon (already used by UnmetDemandCard)
- Remove `TopZeroResultsCard` component entirely
- The merged card goes in position 1 in the new layout

### Anti-Patterns to Avoid

- **Do not change backend API endpoints or response shapes** — this is frontend-only label cleanup
- **Do not change URL routes** (`/admin/gaps` stays as-is for the GapsPage)
- **Do not merge the stat card `gap_count` with `/events/demand` data** — they measure different things; only rename the label
- **Do not resize cards** — ordering creates visibility, not size changes (user decision)
- **Do not add breakdown badges** to merged cards (user decision: "merged cards are clean and simple")

## Common Pitfalls

### Pitfall 1: Breaking the "See all" link on Zero-Result Queries card
**What goes wrong:** `TopZeroResultsCard` has a `Link to="/admin/gaps"` — this belongs on the merged card.
**Why it happens:** `UnmetDemandCard` doesn't have this link. Easy to copy only UnmetDemandCard's structure.
**How to avoid:** The merged `ZeroResultQueriesCard` must include the `Link to="/admin/gaps"` in the header, same as `TopZeroResultsCard`.

### Pitfall 2: Merged card losing period toggle awareness
**What goes wrong:** `TopZeroResultsCard` had hardcoded `days: 30`. If implementer uses that as the base, the merged card won't respect the period toggle.
**How to avoid:** Base the merge on `UnmetDemandCard` (which already takes `days` prop). Ensure the merged component receives `days` from the parent `OverviewPage`.

### Pitfall 3: Forgetting the stat row label
**What goes wrong:** The 4 stat mini-cards at the top include `label="Gaps"` — easy to miss since it's in a different JSX block from the detail cards.
**How to avoid:** Search for `"Gaps"` in OverviewPage.tsx — appears on line 480. Must be renamed to `"Zero-Result Queries"`.

### Pitfall 4: TopExpertsCard title says "Top Experts" not "Top Clicks"
**What goes wrong:** Per the target order spec, position 3 is "Top Clicks". The current `TopExpertsCard` shows clicks-per-expert ranked, titled "Top Experts". The user spec says the Clicks terminology stays as-is ("Top Clicks" and "Recent Clicks" are fine).
**How to avoid:** Rename `TopExpertsCard` title from "Top Experts" to "Top Clicks" to match the order spec terminology. The component still renders the same data.

### Pitfall 5: Grid layout mismatch after flattening 2+3 column grids into one
**What goes wrong:** Current layout uses a 2-col grid for detail cards and a 3-col grid for ranked insights. After merge (removing UnmetDemandCard, promoting ZeroResultQueries), the grid has 6 cards that should flow in order.
**How to avoid:** Use a single 2-col grid for all 6 cards. The new card order is: ZeroResultQueries, TopSearches, TopClicks, RecentSearches, RecentClicks, Leads. In a 2-col grid this pairs as: (ZRQ, TopSearches), (TopClicks, RecentSearches), (RecentClicks, Leads).
