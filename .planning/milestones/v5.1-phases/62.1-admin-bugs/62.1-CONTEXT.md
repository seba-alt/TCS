# Phase 62.1: Admin Bugs - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix naming inconsistencies, remove duplicate cards, and reorder admin dashboard cards for better visibility. No new features — consolidate and clean up existing admin overview.

</domain>

<decisions>
## Implementation Decisions

### Naming Unification
- "Unmet demand", "gaps", and "zero-result queries" all refer to the same concept → unify to **"Zero-Result Queries"** everywhere
- "Leads" and "newsletter leads" are the same → unify to **"Leads"**
- All search types (tag clicks, search bar queries, filter selections) → unify to **"Searches"**
- Search source (tag/bar/filter) kept as metadata in the database, just not surfaced as separate categories in the UI
- Naming changes apply to **both frontend labels AND backend API route names/responses**

### Card Visibility & Order
- "Top" cards (aggregated/ranked) get more visibility than "Recent" cards (chronological)
- Visibility achieved through **ordering** (top cards appear first), not card size differences
- Card order on admin overview:
  1. Zero-Result Queries (prominent, near top — actionable business intel)
  2. Top Searches
  3. Top Clicks
  4. Recent Searches
  5. Recent Clicks
  6. Leads (after searches/clicks)

### Duplicate Card Removal
- Remove duplicate cards that show the same data under different names
- Merged cards are clean and simple — no extra badges or breakdowns
- "Leads" and "newsletter leads" are confirmed same data source → merge to single "Leads" card
- "Zero-result queries" / "unmet demand" / "gaps" data sources need investigation (user unsure if same backend source)
- Search type separation (tag/bar/filter) needs investigation (user unsure of current implementation)

### Claude's Discretion
- How to handle backend data source merging if gaps/ZRQ pull from different sources
- Exact card layout and spacing after reorder
- How search types are currently separated (investigate and merge as needed)

</decisions>
