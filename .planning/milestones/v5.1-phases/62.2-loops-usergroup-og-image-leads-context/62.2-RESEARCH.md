## Summary

This phase has three independent, narrowly scoped features that can be implemented in parallel.

**Feature 1 — Loops userGroup:** The `userGroup` field is a built-in Loops standard property (confirmed via official docs). The existing `sync_contact_to_loops()` function in `app/loops.py` only needs one line added to its payload dict. The newsletter subscriber endpoint (`app/routers/newsletter.py`) does NOT currently call Loops — only `app/routers/email_capture.py` does. Both call sites must be wired to pass `userGroup: "search"`. No schema changes, no new libraries.

**Feature 2 — OG image:** The `frontend/index.html` already has partial OG tags but with `twitter:card=summary` (not `summary_large_image`) and the existing `og-image.png` (1000x994, 12KB square) differs from the new `OG picture.png` (1200x627, 175KB — correct 1.91:1 ratio). The task is: copy new image to `frontend/public/og-image.png` (overwrite), update `twitter:card` to `summary_large_image`. The Vite SPA serves a single `index.html` so all routes share the same tags — the decision to apply to "all pages" is automatically satisfied by the single HTML file.

**Feature 3 — Leads timeline session linking:** This is the most involved change. The `NewsletterSubscriber` model lacks a `session_id` column. The `newsletter.py` subscribe endpoint doesn't accept `session_id`. The frontend `handleSubscribe()` in `MarketplacePage.tsx` doesn't pass session_id. The `user_events` table already has `session_id` indexed. The `get_lead_timeline()` endpoint already queries `Conversation` by email for search events — but the search events in `Conversation` are pre-email-gate (logged under email from the chat flow). The unlinked anonymous searches are in `user_events` with `event_type='search_query'`. To bridge them: capture `session_id` at newsletter signup, store on `NewsletterSubscriber`, then query `user_events WHERE session_id = ?` to pull anonymous pre-signup searches into the timeline.

**Primary recommendation:** Implement as three separate tasks in a single plan wave. Features 1 and 2 are trivial (~10 min each). Feature 3 requires a DB schema change (new nullable column + SQLite migration via `create_all`) + backend endpoint update + frontend changes in 3 files.

### Feature 3: Leads Timeline Session Linking

**Architecture:** Anonymous sessions bridge to identified leads via `session_id`.

**Data flow:**
```
Frontend localStorage → tcs_session_id (UUID)
  ↓ (at newsletter subscribe)
POST /api/newsletter/subscribe { email, session_id }
  ↓
NewsletterSubscriber.session_id (nullable String(64))
  ↓ (at timeline fetch)
GET /api/admin/lead-timeline/{email}
  ↓ queries user_events WHERE session_id = subscriber.session_id
  ↓ merges with existing Conversation search events + LeadClick events
  ↓ returns unified timeline
```

**Schema change** (`app/models.py`):
```python
class NewsletterSubscriber(Base):
    # ... existing fields ...
    session_id: Mapped[str | None] = mapped_column(String(64), nullable=True, index=True)
```

Note: `nullable=True` means SQLAlchemy's `create_all` will add this column without errors on existing Railway DB (SQLite `ALTER TABLE ADD COLUMN` semantics).

**Backend request model** (`app/routers/newsletter.py`):
```python
class NewsletterSubscribeRequest(BaseModel):
    email: EmailStr
    source: str = "gate"
    session_id: str | None = None   # ADD
```

**Store session_id in subscribe handler:**
```python
stmt = (
    insert(NewsletterSubscriber)
    .values(email=str(body.email), source=body.source, session_id=body.session_id)
    .on_conflict_do_nothing(index_elements=["email"])
)
```

**Timeline endpoint** (`app/routers/admin/leads.py` — `get_lead_timeline`): Add a third event source:
```python
# After fetching click_rows, fetch user_events for the linked session
subscriber = db.scalar(
    select(NewsletterSubscriber).where(NewsletterSubscriber.email == email)
)
if subscriber and subscriber.session_id:
    event_rows = db.scalars(
        select(UserEvent).where(
            UserEvent.session_id == subscriber.session_id,
            UserEvent.event_type == "search_query"
        )
    ).all()
    for row in event_rows:
        payload = json.loads(row.payload or "{}")
        search_events.append({
            "type": "search",
            "query": payload.get("query_text", ""),
            "result_count": payload.get("result_count", 0),
            "created_at": row.created_at.isoformat(),
        })
```

**Frontend: `getSessionId` already exists in `tracking.ts`** — reads from `localStorage.getItem('tcs_session_id')`. This is the value to send.

**Frontend: `MarketplacePage.tsx` `handleSubscribe`** needs to read session_id and include it in the POST:
```typescript
import { getSessionId } from '../tracking'  // or inline localStorage.getItem

async function handleSubscribe(email: string) {
  setSubscribed(email)
  setShowGate(false)

  const sessionId = localStorage.getItem('tcs_session_id')

  fetch(`${API_URL}/api/newsletter/subscribe`, {
    method: 'POST',

  }).catch(() => {})
```

Note: `getSessionId()` in `tracking.ts` is NOT exported — it is a module-private function. The planner must either export it or inline `localStorage.getItem('tcs_session_id')` in `MarketplacePage.tsx`. Reading directly from localStorage is simpler and avoids touching `tracking.ts`.

### Anti-Patterns to Avoid
- **Exporting `getSessionId` from tracking.ts:** Unnecessary coupling — just read `localStorage.getItem('tcs_session_id')` directly in the subscribe handler.
- **Backfilling existing NewsletterSubscriber rows with session_id:** Out of scope per user decision.
- **Querying ALL user_events for timeline (no session_id filter):** Would return anonymous data from other users. Always filter by the specific linked session_id.
- **Using `create_all` to drop and recreate table:** Never. SQLite `create_all` with `checkfirst=True` only adds missing tables/columns — won't drop existing data.

## Common Pitfalls

### Pitfall 1: newsletter.py does NOT currently call Loops
**What goes wrong:** Adding `userGroup` to `loops.py` only affects `email_capture.py`. Newsletter subscribers won't appear in Loops with `userGroup: "search"` because `newsletter.py` never calls `sync_contact_to_loops`.
**Why it happens:** The newsletter endpoint was added separately and never wired to Loops.
**How to avoid:** Wire `BackgroundTasks` + `sync_contact_to_loops` into `newsletter.py` subscribe handler as part of this phase.
**Warning signs:** Check `newsletter.py` — no import of `sync_contact_to_loops` exists currently.

### Pitfall 2: sqlite INSERT OR IGNORE swallows session_id on duplicate emails
**What goes wrong:** If a user subscribes twice (same email), the second call hits `on_conflict_do_nothing`. The session_id from the second session is silently ignored. The stored session_id stays from the first signup.
**Why it happens:** Idempotent insert design — correct for emails, but means session_id is frozen at first signup.
**How to avoid:** This is acceptable per user decisions ("current session only"). No action needed — document the behavior clearly in code comments.
**Warning signs:** User complains timeline doesn't update after re-signup. This is by design.

### Pitfall 3: user_events payload key is `query_text` not `query`
**What goes wrong:** Timeline endpoint reads `payload.get("query", "")` but the actual key stored by `trackEvent('search_query', { query_text: ... })` is `query_text`.
**Why it happens:** Different field naming between Conversation.query and user_events payload.
**How to avoid:** Use `payload.get("query_text", "")` when parsing user_events payload for timeline. Confirmed in `useExplore.ts` line 90: `query_text: query`.
**Warning signs:** Timeline shows empty string for all session-linked search events.

### Pitfall 4: OG image aspect ratio mismatch with twitter:card type
**What goes wrong:** `twitter:card=summary` uses square images (144x144–4096x4096). `twitter:card=summary_large_image` uses landscape (1.91:1, min 300×157). Mixing the wrong card type with a landscape image causes poor cropping.
**Why it happens:** Existing `og-image.png` was 1000x994 (square) — designed for `summary`. New image is 1200x627 — designed for `summary_large_image`.
**How to avoid:** Change `twitter:card` to `summary_large_image` when replacing the image file.
**Warning signs:** Twitter/X shows a small square thumbnail instead of large banner.

### Pitfall 5: `getSessionId()` is not exported from tracking.ts
**What goes wrong:** Attempting to import `getSessionId` from `tracking.ts` gives a TypeScript error — it's a module-private function.
**Why it happens:** `getSessionId` was designed as internal utility only.
**How to avoid:** Read `localStorage.getItem('tcs_session_id')` directly in `MarketplacePage.tsx`. The key `'tcs_session_id'` is defined as `SESSION_KEY` constant in `tracking.ts`. Can inline the string literal or define a local const.
**Warning signs:** TypeScript compiler error on import.

## Open Questions

1. **Does `create_all` with `checkfirst=True` add columns on Railway's SQLite instance?**
   - What we know: SQLAlchemy `Base.metadata.create_all(checkfirst=True)` runs `CREATE TABLE IF NOT EXISTS`. It does NOT automatically run `ALTER TABLE ADD COLUMN` for existing tables with new columns.
   - What's unclear: Railway's SQLite instance may already have `newsletter_subscribers` without `session_id`. Pure `create_all` will NOT add the column.
   - Recommendation: Use a startup migration snippet in `app/database.py` or `app/main.py`:
     ```python
     # Safe: SQLite ignores "duplicate column" error via try/except
     try:
         engine.execute("ALTER TABLE newsletter_subscribers ADD COLUMN session_id VARCHAR(64)")
     except Exception:
         pass  # column already exists
     ```
     Or check for the column first. This is the established pattern for schema changes in this project (no Alembic).

2. **Does the `on_conflict_do_nothing` insert affect session_id on re-subscribe?**
   - What we know: Yes — on duplicate email, the entire insert is skipped, so session_id from a later session won't update the record.
   - Recommendation: Document this as intentional. If update-on-re-subscribe is ever needed, switch to upsert pattern. Out of scope for this phase.

3. **Are there other call sites for newsletter subscribe?**
   - What we know: `MarketplacePage.tsx` is the only place that calls `POST /api/newsletter/subscribe` from the frontend (confirmed via grep).
   - What's unclear: None — fully confirmed single call site.
