## Summary

Phase 58.1 is a pure frontend improvement phase ŌĆö no new backend capabilities are needed, but one backend endpoint needs a new response field (`click_count` on leads) and the `analytics-summary` endpoint's `recent_searches` payload needs to surface `active_tags` already stored in `user_events.payload`. All other changes are React component refactors, UI fixes, and layout restructuring.

The core problems are four: (1) Explore search rows appear empty when the user searched by tags only (because `query_text` is `""` and `active_tags` is stored in the event payload but not surfaced in the admin); (2) lead click activity is only visible after expanding individual rows, not at a glance; (3) the Data page has two tabs (Searches / Marketplace) that should be merged into one unified page; (4) "Top Feedback" and "Top Searched Queries" sections on Overview are low-value or broken and need to be evaluated for removal or fix.

The stack is entirely React (Vite + Tailwind + React Router v7 + TanStack Table). No new npm packages are needed ŌĆö recharts is already installed for the existing trend chart, and TanStack Table is already used in `SearchesTable`. The pattern for all UI work is: edit existing page/component files, wire to existing hooks or extend them minimally, keep the dark-Tailwind aesthetic established across the codebase.

**Primary recommendation:** Make the minimum backend change (expose `click_count` on `/leads` + expose `active_tags` in `analytics-summary` recent_searches), then do all remaining work in frontend-only React edits.

## Backend Investigation

### What backend changes are needed

**1. `/leads` endpoint ŌĆö add `click_count` field (HIGH confidence)**

Currently `LeadsPage.tsx` only loads click counts lazily when a row is expanded, via a separate `GET /lead-clicks` call. The leads table has no "Clicks" column. The backend `get_leads()` in `app/routers/admin/leads.py` groups `Conversation` rows but never touches `lead_clicks` table.

Fix: Join or subquery the `lead_clicks` table to count clicks per email in `get_leads()`:

```python
# In app/routers/admin/leads.py ŌĆö get_leads()
# Add a subquery count for click_count per email
from sqlalchemy import func, select
from app.models import Conversation, LeadClick

click_counts = db.execute(
    select(LeadClick.email, func.count(LeadClick.id).label("click_count"))
    .group_by(LeadClick.email)
).all()
click_map = {r.email: r.click_count for r in click_counts}

# Then include in result dict:
result.append({
    ...
    "click_count": click_map.get(lead["email"], 0),
})
```

**2. `/analytics-summary` ŌĆö expose `active_tags` in recent_searches (HIGH confidence)**

The `search_query` event stores `active_tags` in its JSON payload (verified in `useExplore.ts` line 91):

```typescript
void trackEvent('search_query', {
  query_text: query,       // can be "" for tag-only searches
  active_tags: tags,       // array of tag strings
  result_count: ...,
})
```

The backend `get_analytics_summary()` in `analytics.py` currently only reads `query_text` and `result_count`:

```python
SELECT
    json_extract(payload, '$.query_text') AS query_text,
    json_extract(payload, '$.result_count') AS result_count,
    created_at
FROM user_events
WHERE event_type = 'search_query'
```

Fix: Add `json_extract(payload, '$.active_tags')` to the query and include it in the response. The type in `types.ts` also needs updating.

**3. Chronological click activity endpoint ŌĆö evaluate approach**

For the "dedicated click activity view" (chronological table of ALL click events), the existing `GET /lead-clicks` endpoint returns data grouped by email. The frontend can flatten this ŌĆö it's already fetching everything. Alternatively: the endpoint could accept an optional `flat=true` param, or we can re-use `/lead-clicks` and flatten on the frontend.

**Recommendation:** Flatten on the frontend. Avoid a new endpoint for a simple reshape. The `LeadClicksResponse` already contains all the data; just sort and flatten the `leads[].clicks` arrays by `created_at`.

## Common Pitfalls

### Pitfall 1: Tag-only searches appearing as empty rows

**What goes wrong:** `RecentSearchEntry.query_text` is `""` when user searched by tags only. Current template just renders `{row.query_text}` ŌĆö blank.

**Why it happens:** `trackEvent('search_query', { query_text: query, active_tags: tags })` ŌĆö when `query` is empty string, `query_text` is `""`.

**How to avoid:**
1. Backend: extend `analytics-summary` SQL to also extract `json_extract(payload, '$.active_tags')`
2. Frontend: add `active_tags?: string[]` to `RecentSearchEntry` type
3. In `RecentExploreSearchesCard`: if `row.query_text` is empty and `row.active_tags.length > 0`, render chips instead of text. If both empty, skip the row or show "Browse (no filter)".

**Warning signs:** Any `RecentSearchEntry` with `query_text === ""` in the recent_searches array.

### Pitfall 2: DataPage route removal breaking existing links

**What goes wrong:** After removing `DataPage` child routes, existing `<Link to="/admin/data/searches">` or `<Link to="/admin/data/marketplace">` links elsewhere in the codebase produce 404s.

**How to avoid:** Grep for all references to `/admin/data/searches` and `/admin/data/marketplace` before removing routes. In `OverviewPage.tsx`, `RecentExploreSearchesCard` links to `/admin/data/searches` and `RecentCardClicksCard` links to `/admin/data/marketplace`. Update both links to `/admin/data` after merge.

**Checked occurrences:**
- `OverviewPage.tsx` line 178: `to="/admin/data/searches"`
- `OverviewPage.tsx` line 212: `to="/admin/data/marketplace"`
- `SearchesTable.tsx` line 65: `navigate('/admin/leads', ...)` ŌĆö not a data link, fine

### Pitfall 3: Leads table re-renders on every sort state change

**What goes wrong:** Sorting the leads table by `click_count` with local state triggers full re-render including the expanded row state, collapsing any open row.

**How to avoid:** Keep `expandedEmail` and `sortField` in separate state. Sort the data array before rendering; don't mutate it. Use `useMemo` for sorted leads:

```typescript
const sortedLeads = useMemo(() => {
  if (!data) return []
  return [...data.leads].sort((a, b) => sortDir === 'desc' ? b.click_count - a.click_count : a.click_count - b.click_count)
}, [data, sortField, sortDir])
```

### Pitfall 4: Unified date picker not wiring to SearchesPage's filter state

**What goes wrong:** `SearchesPage.tsx` has local `dateFrom`/`dateTo` state inside the component. After merging into `DataPage`, the date picker needs to live at the DataPage level and be passed down as props.

**How to avoid:** Lift the date state to the merged DataPage component. Pass `dateFrom`/`dateTo` as props to the Searches table section and to the TrendSection chart.

### Pitfall 5: `top_queries` section removed ŌĆö stat card "Total Searches" link

**What goes wrong:** The "Total Searches" stat card currently navigates to `/admin/data/searches` on click. After route changes, the path needs updating.

**How to avoid:** When changing routes, update `onClick={() => navigate('/admin/data/searches')}` to `navigate('/admin/data')` in the merged setup.

### Extending `get_leads()` with click_count

# Source: app/routers/admin/leads.py
@router.get("/leads")
def get_leads(db: Session = Depends(get_db)):
    rows = db.scalars(
        select(Conversation).order_by(Conversation.created_at.desc())
    ).all()

    # NEW: Batch count clicks per email
    click_counts_rows = db.execute(
        select(LeadClick.email, func.count(LeadClick.id).label("click_count"))
        .group_by(LeadClick.email)
    ).all()
    click_map = {r.email: r.click_count for r in click_counts_rows}

    leads: dict[str, dict] = {}
    for row in rows:
        email = row.email
        if email not in leads:

                "email": email,

        leads[email]["total_searches"] += 1
        if _is_gap(row):
            leads[email]["gap_count"] += 1
