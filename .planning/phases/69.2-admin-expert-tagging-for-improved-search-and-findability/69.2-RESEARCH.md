## Summary

Phase 69.2 adds a dedicated tag manager to the admin panel that allows admins to curate a predefined list of domain expertise tags and assign them to experts in bulk. The critical architectural tension is preserving manual tags through AI re-tagging cycles. The cleanest solution is a separate `Expert.manual_tags` column (JSON array, analogous to the existing `Expert.tags` column) rather than a `tag_type` discriminator in `expert_tags`, because it makes the "Gemini only touches its own column" invariant trivially enforceable — no query guards needed, no merge logic complexity.

The existing `ExpertTag` normalized table already has a `tag_type` discriminator ("skill" / "industry"). A third type "manual" is addable without schema migration issues. The public-facing search pipeline (`run_explore`) filters `ExpertTag` rows by `tag_type == "skill"` for the clickable tag filter; manual tags need to be included in this filter logic and in FAISS document text so they improve semantic search. The FAISS rebuild is already user-triggered via the existing `POST /api/admin/ingest/run` endpoint — no new trigger mechanism is needed.

The frontend admin pattern is well established: new route + page file in `frontend/src/admin/pages/`, new `NavLink` in `AdminSidebar.tsx`, new data hooks in `useAdminData.ts`, and new backend endpoints registered in `app/routers/admin/__init__.py`. The tag manager page is the largest piece; the bulk-assign workflow is the most complex interaction to design carefully.

**Primary recommendation:** Use a dedicated `Expert.manual_tags` JSON column (separate from `Expert.tags`) for clean AI/manual separation, merged at read time for public display. Register a "manual" tag_type in `ExpertTag` for indexed filtering.

### Pattern 1: Manual Tags Column (Data Model)

**What:** Add `Expert.manual_tags` as a separate JSON Text column, parallel to `Expert.tags` (AI-generated skill tags).

**Why this over a single tags column with source metadata:**
- `tag_experts.py` script writes only to `Expert.tags` — zero risk of overwriting manual tags
- No merge complexity on write — Gemini job never touches `manual_tags`
- Merge happens only at read time (serialization) — simple `list(set(ai_tags + manual_tags))`
- Deduplication is trivially correct: Python set union

**Model change (`app/models.py`):**
```python
class Expert(Base):
    # ... existing fields ...
    tags: Mapped[str | None] = mapped_column(Text, nullable=True)          # AI-generated (Gemini)
    manual_tags: Mapped[str | None] = mapped_column(Text, nullable=True)   # Admin-assigned (NEW)
    industry_tags: Mapped[str | None] = mapped_column(Text, nullable=True)
```

**SQLite migration (in `app/main.py` lifespan, matching existing pattern):**
```python
# Phase 69.2: manual_tags column for admin-assigned tags
with engine.connect() as _conn:
    try:
        _conn.execute(_text("ALTER TABLE experts ADD COLUMN manual_tags TEXT"))
        _conn.commit()
    except Exception:
        pass  # Column already exists — idempotent
```

**Serialization merge in `_serialize_expert()` (`_common.py`):**
```python
def _serialize_expert(e: Expert) -> dict:
    ai_tags = json.loads(e.tags or "[]")
    manual_tags = json.loads(e.manual_tags or "[]")
    # Deduplicate: manual tags that match an existing AI tag appear only once
    merged_tags = list(dict.fromkeys(ai_tags + [t for t in manual_tags if t not in ai_tags]))
    return {
        ...
        "tags": merged_tags,
        "manual_tags": manual_tags,   # expose separately so admin UI can show what's editable
        ...
    }
```

### Pattern 2: Predefined Tag List (TagCatalog table)

**What:** A new `TagCatalog` DB table stores the admin-managed list of available tags. Seeded from unique values in `Expert.tags` at first use.

**Why a DB table (not hardcoded list):**
- Admin can add new tags at any time without code deployment
- Tag list survives restarts
- Consistent with existing `AppSetting` pattern for admin-managed runtime data

**Model addition (`app/models.py`):**
```python
class TagCatalog(Base):
    """Admin-managed predefined tag list (Phase 69.2). Seeded from existing AI skill tags."""
    __tablename__ = "tag_catalog"

    id: Mapped[int] = mapped_column(primary_key=True)
    tag: Mapped[str] = mapped_column(String(200), unique=True, nullable=False, index=True)
    created_at: Mapped[datetime.datetime] = mapped_column(
        DateTime, default=datetime.datetime.utcnow, nullable=False
    )
```

**Seeding strategy:** On first `GET /api/admin/tags/catalog`, if the catalog is empty, auto-seed from `SELECT DISTINCT tag FROM expert_tags WHERE tag_type = 'skill'`. This is a lazy seed — no startup migration needed.

### Pattern 3: ExpertTag Extension (manual tag_type)

**What:** When admin assigns a manual tag to an expert, write a row in `expert_tags` with `tag_type = "manual"` (in addition to writing to `Expert.manual_tags`).

**Why:** The `ExpertTag` table is already the index used by `run_explore()` for tag filtering. The search pipeline does `ExpertTag.tag_type == "skill"` — this needs to be extended to also match "manual" tags. Adding "manual" rows to `expert_tags` keeps the existing query pattern intact.

**Updated `sync_expert_tags()` signature:**
```python
def sync_expert_tags(
    db: Session,
    expert_id: int,
    tags: list[str],           # AI skill tags
    industry_tags: list[str],
    manual_tags: list[str] = [],  # NEW: admin-assigned tags
) -> None:
    db.execute(delete(ExpertTag).where(ExpertTag.expert_id == expert_id))
    rows = [
        ExpertTag(expert_id=expert_id, tag=t.lower().strip(), tag_type="skill")
        for t in tags if t and t.strip()
    ] + [
        ExpertTag(expert_id=expert_id, tag=t.strip(), tag_type="industry")
        for t in industry_tags if t and t.strip()
    ] + [
        ExpertTag(expert_id=expert_id, tag=t.lower().strip(), tag_type="manual")
        for t in manual_tags if t and t.strip()
    ]
    if rows:
        db.bulk_save_objects(rows)
```

**Search pipeline update (`explorer.py`):** Change tag filter from `tag_type == "skill"` to `tag_type.in_(["skill", "manual"])` so public tag-filter clicks match on both AI and admin tags:

### Pattern 6: AI Re-tagging Safety (tag_experts.py)

The `scripts/tag_experts.py` script only writes to `Expert.tags` — it never touches `Expert.manual_tags`. This is the key invariant.

**Verification point:** Confirm `tag_experts.py` writes only `expert.tags = json.dumps(tags)` (it does — confirmed in research). No code change needed to `tag_experts.py`.

**`_run_tag_job()` in `_common.py`** also calls `tag_experts.py` as a subprocess — same safety applies.

**Post-tag-job sync:** The `_run_ingest_job` function calls `sync_all_expert_tags()` after tagging. This function must be updated to also populate manual tag rows (see Pattern 3 above).

### Anti-Patterns to Avoid

- **Do NOT add manual tags to `Expert.tags` JSON column.** That column is Gemini's territory — the tag_experts.py script overwrites it on every run. Manual tags written there would be silently destroyed on the next AI tagging run.
- **Do NOT automatically trigger FAISS rebuild on tag assignment.** CONTEXT.md explicitly locks this as manual. The admin clicks "Rebuild Index" from the existing Tools page when ready.
- **Do NOT use free-text tag input.** Predefined catalog only — this is locked.
- **Do NOT allow removing AI/industry tags from the tag manager.** Admin can only add manual tags — this is locked. The delete endpoint must only remove manual_tags rows, never AI skill tag rows.
- **Do NOT create a separate tag_type column per expert row.** The existing `ExpertTag` normalized table with `tag_type` discriminator is already the right design — extend it, don't replace it.

## Common Pitfalls

### Pitfall 1: Manual Tags Overwritten by sync_all_expert_tags at Startup

**What goes wrong:** The startup sequence calls `sync_all_expert_tags()`. If this function is not updated to handle `manual_tags`, it will delete all "manual" ExpertTag rows on every server restart (it deletes all rows then re-inserts from `Expert.tags` and `Expert.industry_tags` only).

**Why it happens:** `sync_all_expert_tags()` does `db.execute(delete(ExpertTag))` — a full table wipe — then re-inserts. Without reading `Expert.manual_tags`, manual rows are lost.

**How to avoid:** Update `sync_all_expert_tags()` to also read `Expert.manual_tags` and insert rows with `tag_type="manual"`.

**Warning signs:** Tags appear correctly after assignment but disappear after the next server restart or after triggering FAISS rebuild (which also calls `sync_all_expert_tags()`).

### Pitfall 2: FAISS Rebuild Does Not Include Manual Tags

**What goes wrong:** Admin assigns manual tags, rebuilds FAISS. The rebuilt index documents don't include manual tag text, so semantic search doesn't benefit from the new tags.

**Why it happens:** `scripts/ingest.py` builds document text from `Expert.tags` (and possibly bio/title). If manual_tags aren't merged into the document text fed to FAISS, the embedding doesn't reflect them.

**How to avoid:** Update `scripts/ingest.py` to merge `Expert.tags` and `Expert.manual_tags` when building the document text for embedding. The merged list approach (deduplicated) is correct.

**Warning signs:** Tag filter works (because ExpertTag is updated) but semantic search ignores manual tags.

### Pitfall 3: ExpertType in ExpertTag Not Filterable

**What goes wrong:** The explorer.py tag filter uses `tag_type == "skill"`. If manual tags are stored as `tag_type="manual"` but the filter still uses `== "skill"`, clicking a manually-added tag on the public site returns 0 results.

**How to avoid:** Change explorer.py to `tag_type.in_(["skill", "manual"])` when filtering by user-selected tags.

### Pitfall 4: Tag Catalog Delete Does Not Unassign from Experts

**What goes wrong:** Admin deletes a tag from the catalog. The expectation might be that all experts lose that tag. However, the locked decision says FAISS rebuild is manual — and clean cleanup requires touching many Expert records.

**How to avoid:** Document this in the UI: "Deleting a tag from the catalog removes it from future assignment options but does NOT retroactively remove it from experts already tagged. Use the Tag → Experts view to manually remove it from each expert."

**This is acceptable behavior** given the admin-only scope and the small scale.

### Pitfall 5: Sidebar Route Conflict

**What goes wrong:** Adding `/admin/tags` to the router, but the React Router config doesn't include a `<Route path="tags">` under the admin layout.

**How to avoid:** Add the route in the same place existing admin routes are defined. From reading the code: `AdminApp.tsx` uses `<Outlet />`, and routes are likely defined in `main.tsx` or a dedicated router config file. Need to check during implementation.

## Code Examples

### Backend: Bulk Tag Assignment Endpoint

```python
# app/routers/admin/tags.py
class BulkAssignBody(BaseModel):
    usernames: list[str] = Field(..., min_length=1)
    tags: list[str] = Field(..., min_length=1)

@router.post("/tags/assign")
def bulk_assign_tags(body: BulkAssignBody, db: Session = Depends(get_db)):
    """Add manual tags to one or many experts. Additive — never removes existing tags."""
    experts = db.scalars(
        select(Expert).where(Expert.username.in_(body.usernames))
    ).all()
    if not experts:
        raise HTTPException(status_code=404, detail="No experts found")

    tags_to_add = [t.lower().strip() for t in body.tags if t.strip()]

    for expert in experts:
        existing = json.loads(expert.manual_tags or "[]")
        merged = list(dict.fromkeys(existing + [t for t in tags_to_add if t not in existing]))
        expert.manual_tags = json.dumps(merged)
        # Sync ExpertTag rows
        ai_tags = json.loads(expert.tags or "[]")
        industry_tags = json.loads(expert.industry_tags or "[]")
        sync_expert_tags(db, expert.id, ai_tags, industry_tags, merged)

    db.commit()
    return {"ok": True, "updated": len(experts)}
```

### Backend: Tag Catalog Seed-on-Empty

```python
@router.get("/tags/catalog")
def get_tag_catalog(db: Session = Depends(get_db)):
    """Return all predefined tags. Seeds from AI skill tags if catalog is empty."""
    count = db.scalar(select(func.count()).select_from(TagCatalog))
    if count == 0:
        # Lazy seed from existing AI skill tags
        existing_tags = db.scalars(
            select(ExpertTag.tag).where(ExpertTag.tag_type == "skill").distinct()
        ).all()
        if existing_tags:
            for tag in existing_tags:
                db.add(TagCatalog(tag=tag))
            db.commit()

    tags = db.scalars(select(TagCatalog).order_by(TagCatalog.tag)).all()
    return {"tags": [t.tag for t in tags]}
```

### Backend: Updated sync_all_expert_tags (startup-safe)

```python
def sync_all_expert_tags(db: Session) -> None:
    db.execute(delete(ExpertTag))
    experts = db.query(Expert).filter(
        (Expert.tags != None) | (Expert.industry_tags != None) | (Expert.manual_tags != None)
    ).all()

    for expert in experts:
        rows: list[ExpertTag] = []
        # AI skill tags
        if expert.tags:
            try:
                for t in json.loads(expert.tags):
                    if t and t.strip():
                        rows.append(ExpertTag(expert_id=expert.id, tag=t.lower().strip(), tag_type="skill"))
            except (json.JSONDecodeError, TypeError):
                pass
        # Industry tags
        if expert.industry_tags:
            try:
                for t in json.loads(expert.industry_tags):
                    if t and t.strip():
                        rows.append(ExpertTag(expert_id=expert.id, tag=t.strip(), tag_type="industry"))
            except (json.JSONDecodeError, TypeError):
                pass
        # Manual tags (NEW)
        if expert.manual_tags:
            try:
                for t in json.loads(expert.manual_tags):
                    if t and t.strip():
                        rows.append(ExpertTag(expert_id=expert.id, tag=t.lower().strip(), tag_type="manual"))
            except (json.JSONDecodeError, TypeError):
                pass
        if rows:
            db.bulk_save_objects(rows)

    db.commit()
```
