# Phase 69.1: CSV Upload Sync - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade the CSV upload flow from upsert-only to full sync: experts in the DB but not in the uploaded CSV are soft-deleted, existing experts are updated (skipping empty fields), and the FAISS index is rebuilt automatically. The admin gets a preview with cherry-pick control before changes apply.

</domain>

<decisions>
## Implementation Decisions

### Deletion behavior
- Soft delete (not hard delete) — add an `is_active` flag (or equivalent) to the Expert model
- Soft-deleted experts are excluded from public search and display
- Soft-deleted experts remain visible in admin panel via a filter/toggle (e.g., "Show inactive")
- When a soft-deleted expert's username appears in a future CSV upload, automatically reactivate them
- Conversations linked to soft-deleted experts are also archived/soft-deleted

### Sync trigger
- Every CSV upload is a full sync — no separate sync action needed
- Upload flow becomes: upload CSV → preview (showing adds, updates, and deletions with names) → confirm → apply
- The preview lists expert names that will be soft-deleted so admin can verify
- Admin can deselect specific experts from the deletion list before confirming (cherry-pick)

### Field update rules
- Empty CSV fields do NOT overwrite existing data — only non-empty values update
- Tags (skill tags, industry tags) are NOT importable from CSV — managed through admin panel only
- Industry tags continue to be auto-derived from job title/bio/company on update

### FAISS/search index
- FAISS index and metadata.json are automatically rebuilt after sync completes
- Soft-deleted experts are excluded from the search index

### Claude's Discretion
- Category field handling on update (auto-generate vs CSV override)
- Preview detail level for updated experts (count vs per-field diff)
- FAISS index exclusion strategy (full rebuild excluding inactive vs filter at query time)
- Index rebuild feedback in sync response
- Error handling if index rebuild fails after successful data sync

</decisions>
