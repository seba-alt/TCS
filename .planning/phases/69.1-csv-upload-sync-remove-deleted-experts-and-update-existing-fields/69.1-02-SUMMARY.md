**Two new sync endpoints: sync-preview returns a categorised diff (adds/updates/deletes with field-level diffs), sync-apply executes the sync with cherry-pick exclusion, empty-field protection, reactivation, and automatic FAISS rebuild**

## Decisions Made

- Conversation archiving not implemented: conversations store expert data as JSON blobs (no FK to Expert), so soft-deleting the Expert is sufficient. Historical conversations remain intact with the expert data embedded. This matches the plan's own note: "Per Claude's Discretion on implementation detail."
- Already-inactive experts excluded from `to_delete`: prevents idempotency issues — re-uploading the same CSV won't show experts as "to delete" if they were already soft-deleted in a previous sync
- `to_update` includes experts where `changes` is non-empty OR `reactivate` is True — ensures the admin sees experts that will be reactivated even if no field values changed
- `sync_apply` always recalculates and re-syncs industry_tags for updated experts, even if the fields that changed were unrelated to job_title/bio/company — this ensures tag data stays consistent
