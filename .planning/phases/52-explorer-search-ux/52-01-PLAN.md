**Backend randomization (EXPL-01):**

In `app/services/explorer.py`, modify the `run_explore` function signature to accept an optional `seed: int | None = None` parameter. In the pure filter mode branch (the `else` block where `is_text_query` is False):

1. Instead of sorting by `findability_score DESC`, implement weighted-random shuffling using the seed:
   - Use `random.Random(seed)` for deterministic shuffling given the same seed
   - Weight the shuffle by findability_score so higher-findability experts tend to appear near the top but with variety. Algorithm: assign each expert a sort key of `-(findability_score or 0) + random.random() * spread_factor` where spread_factor controls how much randomization is applied. Use a spread factor of ~30 (enough to shuffle within similar findability tiers without putting low-findability experts at the very top).
   - When `seed is None`, fall back to the current deterministic findability DESC sort (backward compatible).
   - ALL experts remain in results — low findability just pushed toward the bottom, never excluded.

2. When query is non-empty (text search mode), do NOT randomize — keep existing FAISS+BM25 fusion sort unchanged. This is already the case since text query has its own branch.

In `app/routers/explore.py`, add `seed: int = Query(default=0, ge=0)` parameter and pass it to `run_explore()`. Value 0 means no randomization (backward compatible). Any positive integer triggers seeded randomization.

**Frontend session-stable seed (EXPL-01):**

In `frontend/src/hooks/useExplore.ts`:

2. Add `seed` to the fetch params in both the main effect and `loadNextPage` — only when `query` is empty (no text search). When query is non-empty, omit the seed param so backend uses relevance ranking.
3. When user clears search and returns to empty query, the same seed from page load is reused, giving session-stable ordering per CONTEXT.md decision.
